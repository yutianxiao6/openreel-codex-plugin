# 图片生成调用链六维审计

目标场景：用户在 Codex 中要求生成一张图片并直接放到当前 OpenReel 画布，且没有明确指定 OpenReel 模型。默认后端是 Codex 内置图片生成。

## 优化后的稳定路径

1. 每个 Codex 会话首次使用 OpenReel 时完成连接和会话选择；后续图片复用该状态。
2. Codex 内置图片生成调用 1 次，得到一个本地最终图片文件。
3. `openreel_publish_generated_image` 调用 1 次。
4. 新版 OpenReel 用 1 个 multipart 请求完成节点创建、图片保存、产物登记和最终事件广播。插件不调用 OpenReel 图片 provider。

## 六个维度

| 维度 | 审计结论 | 优化前常见开销 | 优化后目标 |
| --- | --- | --- | --- |
| 1. 模型生成 | 一张最终图片只需要一次真正的图片生成。 | 可能先建 OpenReel 节点，再触发 OpenReel prompt/provider，形成额外模型调用。 | Codex 内置图片生成 1 次；OpenReel 模型调用 0 次。 |
| 2. Codex 工具 | 建空节点、上传、再读取验证会增加上下文和失败点。 | `image_gen` 之外再调用 create、upload、get，共 3 次插件工具。 | `image_gen` + publish，共 2 次宿主工具调用；成功后不再 get。 |
| 3. 插件到 OpenReel HTTP | 创建与上传可以由服务端合并为一个公开动作。 | create + upload 为 2 次；若先读合同或解析节点 id 会更多。 | 新版服务 1 次 import-image；旧版首次探测不支持后缓存，随后每张 2 次兼容请求。 |
| 4. 持久化与一致性 | 用户只应看到完成节点，不应看到长期空壳。 | 先提交 idle 节点，再提交媒体结果，失败时可能留下空节点。 | 新接口失败时补偿删除节点；成功只广播 completed 完整节点。 |
| 5. 画布刷新 | 局部壳事件不能代替完整节点内容。 | create/update 事件后重复整图 GET，或只合并局部 payload 导致正文/图片缺失。 | 完整事件带 `snapshot_complete`；画布直接落完整节点，不再追加整图 GET。 |
| 6. 失败与重试 | 重试必须复用已生成文件，不能重复花费图片生成。 | 整条链路重跑会重复出图或产生重复节点。 | 生成失败只重试 image_gen；上传失败复用返回 node_id 和原文件，只重试 upload。 |

## 调用次数预算

不计每个会话只发生一次的连接、列表和选择：

- 正常单图：1 次 Codex 图片生成 + 1 次插件 publish + 1 次 OpenReel 导入 HTTP + 1 次最终 SSE 事件。
- 明确选择 OpenReel provider：1 次 create 插件调用 + 1 次 run 插件调用；内部合同预检和 provider 请求属于这条显式路径。
- 上传兼容回退：新版接口首次返回 404/405 后缓存“不支持”，当前图片再执行 create + upload；同一插件进程后续图片直接两步，不重复探测。
- 成功后不调用 get/canvas 验证；publish 返回的完整 completed 节点就是验证依据。只有返回不完整、失败或后续操作依赖精确画布拓扑时才读取。

显式 OpenReel provider 路径保留 create 与 run 两个宿主工具调用是合理的：create 负责持久化并验证动态节点合同，run 才产生明确的模型费用和媒体任务，合并成黑盒会模糊失败边界与费用动作。插件已经对服务端 UUID 做直通解析，等待或上传拿到内部 UUID 时不再额外调用 `node.get`；可见编号（如 `#3`）仍只解析一次，以保证定位正确。

## 选择规则

- 未指定后端：Codex 内置图片生成优先。
- 明确说“使用 OpenReel 模型/配置的模型/provider”：走 OpenReel 节点合同、创建和运行。
- 明确要求两种结果比较：两个后端各生成一次，分别形成节点；不把一次生成结果伪装成两个后端结果。
