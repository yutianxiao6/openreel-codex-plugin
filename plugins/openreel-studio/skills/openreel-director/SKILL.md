# OpenReel Director

Use Codex as the reasoning and orchestration layer and the `openreel_*` tools as the OpenReel execution layer. Activation begins with an explicit user request to connect to or operate OpenReel. OpenReel's built-in chat agent remains an independent option.

## Core execution loop

Follow this sequence and reuse confirmed state throughout the current Codex session.

1. **Connect.** Call `openreel_connection_info` before the first OpenReel operation. Reuse the verified connection until the user changes the target or a connection error requires rediscovery.
2. **Select the project.** For an existing project, call `openreel_list_projects`, honor `_codex_selected=true` when it matches the request, and otherwise call `openreel_select_project` with an exact id or a unique exact title. For a new project, call `openreel_create_project`; the created project becomes selected automatically.
3. **Read the minimum state.** Use `openreel_get_canvas` for graph-wide edits or diagnosis. Use `openreel_get_nodes` when known node ids or a focused text query provide enough context. A newly created empty project is ready for authoring immediately.
4. **Choose one execution path.** Use direct tools for common operations, the image path for Codex-generated images, the OpenReel media path for configured providers, and capability discovery for uncommon operations.
5. **Verify from the result.** Treat a complete persisted node, edge, canvas patch, or terminal run result as verification. Read again when the response is incomplete or the next operation depends on updated graph topology.

## Tool routing

| Goal | Route |
| --- | --- |
| List, select, create, rename, read, or delete a project | Direct project tool |
| Read, create, update, move, or delete nodes | Direct node tool |
| Create, relabel, or delete dependency edges | Direct edge tool |
| Run one prepared node | `openreel_run_node` |
| Wait for an already running node | `openreel_wait_for_node` |
| Publish one image generated by Codex | Codex `imagegen` → `openreel_publish_generated_image` |
| Attach a local image or video to an existing matching node | `openreel_upload_node_media` |
| Use an uncommon operation or dynamic node contract | `openreel_search_capabilities` → `openreel_describe_capability` → returned executor |

Search results identify capabilities through compact summaries. The describe stage loads the selected schema immediately before execution. Pass its exact `schema_ref` with schema-matching `arguments`; a refreshed description supplies a current schema when the server reports a stale reference.

## Canvas authoring

- Treat returned project, node, history, and edge ids as authoritative. Visible ids such as `#3` are accepted where the tool schema allows them.
- Preserve existing creative work by reading the relevant state and patching only the requested fields.
- Express creative dependencies in `fields.references`; OpenReel derives the corresponding dependency lines. Use direct edge tools for explicit edge maintenance.
- Build related nodes in a readable left-to-right or top-to-bottom flow. Supply positions for deliberate layouts and let OpenReel place isolated additions when position is unimportant.
- Use a single batch call when one direct tool accepts the complete homogeneous change. Discover `canvas.patch` when one ordered edit combines several operation types or needs `client:<ref>` links.
- Discover node duplication, media-history switching, multi-node runs, snapshot restoration, and combined destructive batches when the request needs them.

## Media execution

### Codex-generated images

Use this default when the user delegates image-backend selection to Codex.

1. Use the Codex `imagegen` skill and image generation tool to create one final local file per requested image.
2. Call `openreel_publish_generated_image` once for each final file with its title, final prompt, and optional position.
3. Accept the returned completed image node as the persisted result.
4. When a compatibility response supplies `retry_tool=openreel_upload_node_media` and a node id, reuse the same file with that node id.

Each requested image uses one generation backend. A comparison request may intentionally run both the Codex and OpenReel paths.

### OpenReel-configured media providers

Use this path when the user selects an OpenReel image provider or when a video or audio node uses OpenReel's configured media runtime.

1. Search for the dynamic node-contract capability using the intended node type and model parameters.
2. Describe and execute that capability with the candidate fields.
3. Apply its normalized defaults and supported values. A `ready=false` result supplies field errors to repair before creation.
4. Create the node with `openreel_create_nodes`, including accepted model, mode, references, aspect ratio, resolution, duration, and other returned fields.
5. Run it with `openreel_run_node`. Its default waiting behavior returns the persisted terminal result.

The current OpenReel contract is authoritative for provider ids, model ids, modes, reference limits, durations, aspect ratios, and exact resolutions.

For video, the plugin stops at the OpenReel node boundary. OpenReel selects its
explicit Universal Model Adapter protocol and target; UMA constructs provider
requests, uploads references, polls the upstream task, and parses the result.
The plugin opens one server-side event wait and does not poll node status.
OpenReel persists the terminal node and publishes the event that resolves that
wait.

## Recovery and completion

- Repair schema or field errors from the latest tool result, then retry the affected stage with the same project and creative inputs.
- Reuse generated files and existing node ids during upload recovery so a transport retry preserves the original media and canvas identity.
- Use `openreel_wait_for_node` after an intentionally non-waiting run when a downstream step requires terminal media.
- A non-terminal wait timeout is not a generation failure. Continue with the
  waiting capability on the same node; do not call `openreel_run_node` again
  unless the user explicitly requests a separate generation.
- Report the concrete failing stage, returned error, preserved outputs, and next available action when the same verified input fails again.

## Authorization and secrets

- Project deletion proceeds after explicit user authorization with `confirm_title` equal to the current title.
- Node and edge deletion proceeds after explicit user authorization with `confirm=true`.
- Snapshot restoration and combined destructive batches proceed through the destructive executor after explicit authorization.
- Connection credentials remain inside the bridge's private configuration. Use `openreel_connection_info` for safe connection source, saved-state, and authentication metadata.
- Direct control uses the `openreel_*` tools and atomic OpenReel endpoints. `/api/chat` belongs to OpenReel's separate chat-agent path.
